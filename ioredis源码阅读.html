依赖部分功能

@ioredis

建立了几个函数用于联系 redis 的所有指令。总共有约 240 个指令以 json 形式存储在@ioredis/commands//build/ioredis.json中

json 格式如下:

"acl": {
        "arity": -2,
        "flags": [],
        "keyStart": 0,
        "keyStop": 0,
        "step": 0
    }

list 属性包含所有的redis指令。exist(string)用于判断list中是否存在传入的指令名。hasFlag(string2,string) 用于判断 string1 对应的方法是否存在名为 string2 的flag。getKeyIndexes(commandName, args, options) 获取命令参数中键的索引。

cluster-key-slot

一个 redis 集群包含 16384 个哈希槽（hash slot），数据库中的每个数据都属于这 16384 个哈希槽中的一个。集群使用公式 CRC16(key) % 16384 来计算键 key 属于哪个槽。集群中的每一个节点负责处理一部分哈希槽。

cluster-key-slot封装了generate和generateMulti

  • generate(string) 将字符串（键）转为UTF8然后计算槽哈希值并返回
  • generateMulti(array) 将遍历字符串组应用到 generate 方法上，若组内的槽哈希值均相等则返回槽哈希值，存在不相等则返回-1

debug

Javascript 调试工具

denque

JS 的队列实现。

peek(alias get)在队列中查找指定的的 Index 对应值。不删除

size(alias length)返回队列当前的长度

unshift/push添加元素到队列头/尾部

shift/pop返回并删除队列头/尾部的元素

...

lodash.defaults / lodash.isarguments

lodash 就一集成工具集。Lodash Documentation

ms

时间转换工具。将各种格式的时间（字符串）转为毫秒。使用：ms(string)

redis-error

提供了全部的 Redis Error 类型（因为 redis 的 error 均为自建子类），方便输出及调试。

  • RedisError

    • ReplyError: redis-server的错误
    • ParserError: 解析错误，node对redis返回流的解析错误
    • AbortError: 命令拒绝
    • InterruptError: 中断错误

详细的 properties 还得看 npmjs文档

redis-parser

RESP（redis协议报文）解析器

包含options如下。像几乎所有的模块一样，options 受到原型影响

  • returnReply: function; mandatory
  • returnError: function; mandatory
  • returnFatalError: function; optional, defaults to the returnError function
  • returnBuffers: boolean; optional, defaults to false
  • stringNumbers: boolean; optional, defaults to false

standard-as-callback

高性能标准库，在 promise 上实现 promise 和 callback 双回调。

const asCallback = require('standard-as-callback')

const promise = new Promise(function (resolve) {
  setTimeout(function () {
    resolve('hello world!')
  }, 3000)
})

console.log(asCallback)

asCallback.default(promise, function callback (err, res) { //文档里没有default可能是因为commonjs的关系
  console.log(err, res) // null, 'hello world!'
})

可以看到 callback 的回调在 promise resolve 之后

Index.ts

整个模块的入口文件，基本上着 export require from 导入导出的操作

除了模块导入导出还有下面这一段：

Object.defineProperty(exports, "Promise", {
  get() {
    console.warn(
      "ioredis v5 does not support plugging third-party Promise library anymore. Native Promise will be used."
    );
    return Promise;
  },
  set(_lib: unknown) {
    console.warn(
      "ioredis v5 does not support plugging third-party Promise library anymore. Native Promise will be used."
    );
  },
});

在 exports 对象上定义了 promise 属性，get 会打印 warning 并返回一个原生的 Promise，set 仅打印 warning。作用就是限定 ioredis v5 不再支持第三方 promise。

Redis.ts

Redis.ts 主要实现了Redis-connector对象的创建。

在 Redis 连接类型选择上：

    if (this.options.Connector) {
      this.connector = new this.options.Connector(this.options);
    } else if (this.options.sentinels) {
      const sentinelConnector = new SentinelConnector(this.options);
      sentinelConnector.emitter = this;

      this.connector = sentinelConnector;
    } else {
      this.connector = new StandaloneConnector(this.options);
    }

优先选择自定义连接类，然后是检测option.sentinels选用哨兵模式。为设定则为Standalone独立连接

Redis 类中定义了 options 的解析，以及判断命令是否合法、是否在该对象的连接状态中可用

Redis 的连接中创建了offlineQueue和commandQueue两个队列。

  • commandQueue是当前正在处理的命令队列。适配于 redis 单线程的特性，commandQueue队列顶部发出，队列底部推入，将redis-server返回的数据流与发送的数据解析后一一对应。

  • offlineQueue是可选选项。当检查到redis-server的状态不可写后会将命令推入offlineQueue中，等待循环执行的_readyCheck返回当前状态为ready即可写时推入到commandQueue中执行。若给定的 options 禁用了offlineQueue，则会在当前命令上reject一个错误。

    剩下的就是一些工具方法。其中包含各种模式和状态的处理，以及封装了一些工具函数，隐蔽处理掉了手动 disconnect 的错误。

在接口定义上设置了对各种事件的监听。

Command.ts

Command.ts完成了对单个命令对象的类定义。

单个 Command 对象的 options 只有replyEncoding，errorStack,keyPrefix,readOnly，Command 为这些配置选项分别内建了调用和处理方法。

在构造器上Command.ts对参数做了简单的处理，然后针对给定的keyPrefix做了一些类型转换、Index 设定的简单处理。以及调用了如下的私有方法新建 Promise。

  private initPromise() {
    const promise = new Promise((resolve, reject) => {
      if (!this.transformed) {
        this.transformed = true;
        const transformer = Command._transformer.argument[this.name];    // 针对特殊方法进行类型转换的函数
        if (transformer) {
          this.args = transformer(this.args);
        }
        this.stringifyArguments();
      }

      this.resolve = this._convertValue(resolve);    // _convertValue从buffer转为目标编码
      if (this.errorStack) {
        this.reject = (err) => {
          reject(optimizeErrorStack(err, this.errorStack.stack, __dirname));
        };
      } else {
        this.reject = reject;
      }
    });

    this.promise = asCallback(promise, this.callback);    // 待上方创建的 Promise resolve或reject掉后执行给定的callback
  }

特殊传入（指目的要流向redis-server的）方法包含mset、msetnx、hset、hmset有相应的参数特殊处理方法。

特殊传出（指从redis-server传出的）方法有hgetall，处理上可以看到一个已经解决的 污染问题。

总的来说，在流处理上，command 类对参数和配置选项进行了报文预处理，然后toWritable方法将预处理后的转为合法的RESP协议报文。

utils/Commander.ts

utils/Commander.ts仍然是创建了一个 commander 类，主要功能就是将@ioredis/command提供的方法列表通过遍历list执行generateFunction转化为具体的函数，并将通过lua脚本自定义的扩展方法传唤到generateScriptFunction也添加到命令中

Script.ts

Script.ts是一个针对自定义 lua 脚本的类封装。

在构造器上计算了lua脚本的sha1，然后创建了一个WeakSet记录在该套接字中是否已经发送过。

没发送过的话就是eval指令，发送过的话就是usePipeline的evalsha指令。

ioredis的命令封装由Commnad.ts，utils/Commander.ts，Script.ts就基本实现了。本以为generateFunction会是很神奇的处理，结果就是参数排列组合、编码转换来构造报文。。因为指令都是由redis-server实现的。

另外关于这个通过 lua 脚本自定义命令的地方，使用redis-server用来支援更高效的原子操作的。

lua脚本在redis-server的一个sandbox被执行，环境中的可以变量和方法均已在 这里 给出。然后找了下有没有存在什么 sandbox escape ，只有 ubuntu/debian 打包遗留 redis package 的漏洞：CVE-2022-0543，以及依赖于 collectgarbage 方法的两个 UAF 洞：lua-5.4.4-sandbox-escape/x64/exploit.lua at main · Lua-Project/lua-5.4.4-sandbox-escape · GitHub 、erezto/lua-sandbox-escape (github.com)

collectgarbage 在该沙盒环境中还是存在的。上面两个漏洞还没有在 redis 下试过。

难怪 redis 未授权基本只能靠写马

刚好在写这一部分的时候 Redis 发布了 Redis Security Notice: Heap Overflow Vulnerabilities | Redis，其中 CVE-2022-24834 就是通过 lua 脚本实现的堆溢出。不会二进制的无奈

Datahandler.ts

DataHandler 是一个比较另类的类的写法，因为使用时就直接 new 了但并没有接收返回值。

在构造函数中，就做了两件事，一个是实例化了一个 RedisParser 对象，另一个就是监听了 redis.stream.on('data') 事件，也就是我们在实例化 Redis 时传递过来的 socket client，在 data 事件触发时调用 RedisParser.execute 来完成解析。

目前可以认为在调用 execute 方法后会调用实例化时传入的 return Reply 就可以了，这是一个解析后的 response，我们会拿到这个 response 之后会从 commandQueue 中依次取出之前传入的对象。

取出的方式是按照队列的方式来取的，通过 shift，每次取出队列中的第一个元素。

然后调用元素中 command 属性的 resolve 方法，也就是我们在调用各种 Redis 命令时传入的 callback 了。
